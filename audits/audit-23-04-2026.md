## SECTION 1 — Sécurité (SEC)

### SEC-001 — Couverture de tests quasi-nulle
- **Sévérité :** critical
- **Statut :** IMPROVED (mais reste critique)
- **Constat :** Un seul projet de test existe désormais (`tests/InfraFlowSculptor.GenerationParity.Tests`) couvrant uniquement la parité Bicep/Pipeline contre des fixtures dorées. Aucun test unitaire Domain (invariants), aucun test Application (handlers/validators), aucun test d'intégration EF/PostgreSQL.
- **Risque :** Toute refacto critique (squash de migrations, refonte BicepGenerationEngine, fix DDD-001) reste sans filet.
- **Recommandation :** Ajouter au minimum `Domain.UnitTests` (Value Objects + invariants AzureResource), `Application.UnitTests` (handlers d'autorisation et de validation), `Infrastructure.IntegrationTests` (Testcontainers PostgreSQL). Cible 60 % avant refonte de `BicepGenerationEngine`.

### SEC-002 — Headers de sécurité HTTP toujours absents
- **Sévérité :** critical
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Program.cs](src/Api/InfraFlowSculptor.Api/Program.cs)
- **Constat :** Aucun middleware n'écrit `X-Frame-Options`, `X-Content-Type-Options`, `Strict-Transport-Security`, `Content-Security-Policy`, `Referrer-Policy`. La pipeline reste : `UseCors → UseErrorHandling → UseHttpsRedirection → UseRouting → UseRateLimiter → ...`.
- **Risque :** Clickjacking, MIME sniffing, downgrade HTTPS, exfiltration via referrer.
- **Recommandation :** Ajouter `app.Use((ctx, next) => { ctx.Response.Headers["X-Frame-Options"] = "DENY"; ... })` ou `NetEscapades.AspNetCore.SecurityHeaders` avant `UseHttpsRedirection`. Activer HSTS en prod (`app.UseHsts()`).

### SEC-003 — Rate limiting limité au seul endpoint Login
- **Sévérité :** critical
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/RateLimiting/RateLimiting.cs](src/Api/InfraFlowSculptor.Api/RateLimiting/RateLimiting.cs)
- **Constat :** Une seule policy `Login` (3 req / 10 s). Les ~160 endpoints CRUD et les endpoints lourds de génération (`/projects/{id}/generate-bicep`, `/generate-pipeline`) n'ont aucun rate limiting.
- **Risque :** DoS, abuse de la génération Bicep (CPU intensif, écriture blob storage), facturation Azure non bornée.
- **Recommandation :** Ajouter `GlobalLimiter` (par IP/userId, 100 req/min) + policy `Expensive` sur les endpoints de génération (10 req/min). Appliquer via `.RequireRateLimiting("Expensive")` sur les groupes.

### SEC-004 — Path traversal sur les endpoints `{*filePath}`
- **Sévérité :** critical
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Controllers/ProjectController.cs](src/Api/InfraFlowSculptor.Api/Controllers/ProjectController.cs#L892) (lignes 892, 991, 1084)
- **Constat :** Les routes `GET /{projectId}/generate-bicep/files/{*filePath}`, `/generate-pipeline/files/{*filePath}`, `/generate-bootstrap-pipeline/files/{*filePath}` passent `filePath` directement aux handlers sans validation. Aucune protection contre `../../../etc/passwd` côté endpoint.
- **Risque :** Lecture de fichiers arbitraires hors du dossier d'artefacts si la couche Application/IO ne valide pas.
- **Recommandation :** Valider en entrée d'endpoint (rejet si `..`, `/` initial, segments vides) **et** en sortie via `Path.GetFullPath(combined).StartsWith(rootPath)`. Centraliser dans un helper `SafeRelativePath`.

### SEC-005 — CORS trop permissif (AllowAnyHeader/Method)
- **Sévérité :** high
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Program.cs](src/Api/InfraFlowSculptor.Api/Program.cs#L16)
- **Constat :** `policy.AllowAnyHeader().AllowAnyMethod().WithOrigins("http://localhost:4200")`. Pas de profil prod différent.
- **Recommandation :** Restreindre méthodes (`WithMethods("GET","POST","PUT","DELETE","PATCH")`) et headers (`WithHeaders("Content-Type","Authorization")`). Externaliser les origines en configuration.

### SEC-006 — `MaxRequestBodySize` non bornée
- **Sévérité :** high
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Program.cs](src/Api/InfraFlowSculptor.Api/Program.cs)
- **Constat :** Aucune `ConfigureKestrel(o => o.Limits.MaxRequestBodySize = ...)`. Avec 140 commandes acceptant des payloads JSON, un attaquant peut envoyer des bodies arbitrairement grands.
- **Recommandation :** Limiter à 5 MB pour les endpoints CRUD, 50 MB pour les imports/uploads.

### SEC-007 — Divulgation potentielle d'erreurs internes
- **Sévérité :** high
- **Statut :** STILL OPEN
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Errors/ErrorHandling.cs](src/Api/InfraFlowSculptor.Api/Errors/ErrorHandling.cs)
- **Constat :** Le handler global retourne un ProblemDetails 500 générique mais **ne loggue pas l'exception**. Il est impossible de diagnostiquer en prod, et un éventuel `UseDeveloperExceptionPage()` activé en dev fuite la stack.
- **Recommandation :** Injecter `ILogger`, logger `LogError(exception, "Unhandled")` avec un `traceId`, retourner ce `traceId` dans le ProblemDetails (sans détail). S'assurer que `IsDevelopment()` est gardé.

### SEC-008 — Provisioning utilisateur silencieux dans un middleware
- **Sévérité :** medium
- **Statut :** NEW
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Common/UserProvisioningMiddleware.cs](src/Api/InfraFlowSculptor.Api/Common/UserProvisioningMiddleware.cs)
- **Constat :** Le middleware appelle `dbContext.SaveChangesAsync(context.RequestAborted)` directement (court-circuite l'`IUnitOfWork`) et insère un nouvel utilisateur à chaque première requête authentifiée sans verrouillage applicatif explicite.
- **Risque :** En cas de race condition (deux requêtes concurrentes), violation d'unicité `EntraId` sans gestion explicite ; couplage Api → DbContext (violation des couches).
- **Recommandation :** Déplacer la logique dans un `IUserProvisioningService` côté Application, gérer `DbUpdateException` sur unicité, capter via une commande MediatR.

### SEC-009 — Endpoint `/alive` et `/health` exposés sans auth ni rate limit
- **Sévérité :** medium
- **Statut :** NEW
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Program.cs](src/Api/InfraFlowSculptor.Api/Program.cs#L94)
- **Constat :** Les health checks sont publics sans rate limit. Un attaquant peut les utiliser comme oracle ou source de DoS.
- **Recommandation :** Restreindre à un range IP interne (Azure Front Door / probe), ou ajouter `.RequireRateLimiting("HealthChecks")`.

### SEC-010 — Politique d'autorisation `IsAdmin` codée en dur dans `Program.cs`
- **Sévérité :** low
- **Statut :** NEW
- **Fichier :** [src/Api/InfraFlowSculptor.Api/Program.cs](src/Api/InfraFlowSculptor.Api/Program.cs#L26)
- **Constat :** `AddPolicy("IsAdmin", policy => policy.RequireRole("Admin"))` — magic strings éparpillées dans tout le projet.
- **Recommandation :** Centraliser dans `AuthorizationPolicyNames` (constantes) et `AppRoles`.

## SECTION 2 — Base de données / EF Core (DB)

### DB-001 — `HasMaxLength` toujours absent sur ~10 propriétés string
- **Sévérité :** critical
- **Statut :** IMPROVED (couverture partielle)
- **Fichiers :** [BlobContainerConfiguration.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/Configurations/BlobContainerConfiguration.cs), [StorageQueueConfiguration.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/Configurations/StorageQueueConfiguration.cs), [StorageTableConfiguration.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/Configurations/StorageTableConfiguration.cs), [RoleAssignmentConfiguration.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/Configurations/RoleAssignmentConfiguration.cs), [ParameterDefinitionConfiguration.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/Configurations/ParameterDefinitionConfiguration.cs)
- **Constat :** `RoleAssignment.RoleDefinitionId` (GUID Azure ~36 chars) sans `HasMaxLength`. Plusieurs `Name` de sous-ressources Storage sans contrainte (limite Azure 63 chars).
- **Risque :** Index inefficaces (PostgreSQL utilise `text` non borné), absence de validation à l'écriture.
- **Recommandation :** Audit final colonne par colonne. Cible : 100 % des `string` ont un `HasMaxLength` aligné sur la limite Azure.

### DB-002 — Index manquants sur colonnes de filtre fréquentes
- **Sévérité :** high
- **Statut :** IMPROVED
- **Constat :** ~22 `HasIndex` désormais présents. Restent absents :

- **Recommandation :** Ajouter `HasIndex(r => r.SourceResourceId)`, `HasIndex(c => c.ProjectId)`. Vérifier la création automatique des FK via `dotnet ef migrations script`.

### DB-009 — `BaseRepository.AddAsync/UpdateAsync` simulent l'async via `Task.FromResult`

- **Constat :** `AddAsync` retourne `Task.FromResult(res.Entity)`, `UpdateAsync` aussi. EF Core `DbSet<T>.AddAsync` n'est utilisé que pour `HiLo`, mais le pattern global pollue la signature et n'accepte pas `CancellationToken`.
- **Recommandation :** Soit méthodes synchrones (`Add`, `Update`), soit aligner sur `DbContext` avec `CancellationToken`.

### DB-010 — Pas de `CancellationToken` sur `AddAsync` / `DeleteAsync` / `UpdateAsync`
- **Sévérité :** medium
- **Statut :** NEW

- **Recommandation :** Aligner toutes les signatures sur `(.., CancellationToken cancellationToken = default)`.

### DB-011 — Configuration EF dispersée entre 75 fichiers sans convention partagée
- **Sévérité :** medium
- **Statut :** NEW
- **Constat :** 75 configurations entièrement libres : pas de classe de base appliquant `HasMaxLength` par défaut, pas de scanner d'enum / value objects.
- **Recommandation :** Introduire un `ApplyConfigurationsFromAssembly` + une `BaseEntityConfiguration<T>` qui enregistre conventions communes (dates UTC, identifiants `ValueGeneratedNever`, `HasMaxLength` selon attribut).

### DB-012 — Aucun `ExecuteUpdateAsync` / `ExecuteDeleteAsync` sur les opérations bulk
- **Sévérité :** medium
- **Statut :** NEW
- **Constat :** Plusieurs handlers itèrent et `Remove` un par un (e.g. `RemoveAppSettingCommandHandler`, `RemoveProjectMember`). EF Core 10 propose `ExecuteDeleteAsync` qui évite le tracking et émet un seul DELETE.
- **Recommandation :** Migrer les opérations `Remove + SaveChanges` mass vers `ExecuteDeleteAsync`.

### DB-013 — `ProjectDbContext` : DbSets explicites au lieu de `Set<T>()`
- **Sévérité :** medium
- **Statut :** NEW
- **Fichier :** [src/Api/InfraFlowSculptor.Infrastructure/Persistence/ProjectDbContext.cs](src/Api/InfraFlowSculptor.Infrastructure/Persistence/ProjectDbContext.cs)
- **Constat :** ~75 `DbSet<>` exposés publiquement. `null!` partout. Tous les repositories utilisent `Context.Set<T>()` de toute façon.
- **Recommandation :** Réduire aux DbSet utilisés par migrations EF / par requêtes typées explicites. Le reste reste accessible via `Set<T>()`.

### DB-014 — Aucun `QueryFilter` global pour soft-delete / scoping multi-tenant
- **Sévérité :** medium
- **Statut :** NEW
- **Constat :** Pas de soft-delete, pas de query filter. Si demain une dimension multi-tenant ou une corbeille est introduite, il faudra réécrire toutes les queries.
- **Recommandation :** Préparer le terrain via `HasQueryFilter` même no-op + interface `ISoftDeletable`.

### DB-015 — Conversions `Guid?` répétées dans 8+ configurations
- **Sévérité :** low
- **Statut :** NEW
- **Constat :** Le pattern `v => v == null ? (Guid?)null : v.Value, v => v.HasValue ? new XId(v.Value) : null` est dupliqué dans `AppSettingConfiguration`, `RoleAssignmentConfiguration`, etc.
- **Recommandation :** `NullableIdValueConverter<TId>` réutilisable.

### DB-016 — Pas de `[ConcurrencyCheck]` ni `RowVersion` sur les agrégats
- **Sévérité :** low
- **Statut :** NEW
- **Constat :** Aucune protection contre les écritures concurrentes sur le même agrégat.
- **Recommandation :** Ajouter `xmin` (PostgreSQL) ou `RowVersion` au moins sur `Project`, `InfrastructureConfig`, `AzureResource`.

### DB-017 — Pas de transaction explicite pour les opérations multi-aggrégats
- **Sévérité :** low
- **Statut :** NEW
- **Constat :** Les handlers reposent uniquement sur `SaveChangesAsync` du `UnitOfWorkBehavior`. Les opérations qui touchent plusieurs DbContext (aucun aujourd'hui mais possible avec `BicepGenerator.Api`) ou qui appellent un service externe au milieu de la transaction (Git push) ne sont pas atomiques.
- **Recommandation :** Documenter la stratégie : tout effet de bord externe **après** `SaveChanges`, jamais dans le handler avant la commit.

## SECTION 5 — API / Contracts (API)

### API-001 — Path traversal non validé côté contrat (cf. SEC-004)
- **Sévérité :** critical
- **Statut :** STILL OPEN
- **Référence :** SEC-004.

### API-002 — Endpoints manquants `.ProducesProblem(401)` / `(403)`
- **Sévérité :** high
- **Statut :** STILL OPEN
- **Constat :** Audit rapide d'OpenAPI : la majorité des endpoints `/projects/*` ont `Produces(401)` mais beaucoup d'endpoints `KeyVaults`, `Storage`, `Cosmos`, `Service Bus` ne déclarent pas systématiquement 401/403.
- **Recommandation :** Helper `.WithStandardResponses<TResponse>()` qui ajoute 200/400/401/403/404.

### API-003 — `[FromRoute] Guid` accepté sans contrainte de format `:guid`
- **Sévérité :** high
- **Statut :** NEW
- **Constat :** Certains endpoints utilisent `{id}` au lieu de `{id:guid}`. Permet à ASP.NET de tenter le binding sur n'importe quoi.
- **Recommandation :** Audit des routes, contraindre tous les Guid.

### API-004 — Contracts (DTO) ne valident pas via DataAnnotations là où `AddValidation()` est activé
- **Sévérité :** high
- **Statut :** NEW
- **Référence mémoire :** `validation` (la solution active `AddValidation()` + `Microsoft.AspNetCore.Http.Validation.Generated`).
- **Constat :** Les DTOs dans `InfraFlowSculptor.Contracts/*/Requests/*.cs` n'ont pas de `[Required]`/`[StringLength]`. La validation source-générée n'a donc rien à enforcer.
- **Recommandation :** Ajouter DataAnnotations sur tous les Requests pour bénéficier du système de validation activé.

### API-005 — Réponses retournent `Guid` au lieu de `string` pour les Ids
- **Sévérité :** medium
- **Statut :** STILL OPEN (vérifier)
- **Constat :** La convention projet (cf. `.github/copilot-instructions.md` règle 8) est `string`. Plusieurs Responses utilisent encore `Guid`.
- **Recommandation :** Audit Mapster + Responses.

### API-006 — Magic strings de routes éparpillés
- **Sévérité :** medium
- **Statut :** NEW
- **Constat :** `MapGroup("/projects")`, `/infra-configs`, `/keyvaults`… littéraux. Pas de `RouteConstants`.
- **Recommandation :** `static class Routes { public const string Projects = "/projects"; ... }`.

### API-007 — Controllers d'API sont des `static class` avec `UseEndpoints` deprecated
- **Sévérité :** medium
- **Statut :** NEW
- **Constat :** `app.UseEndpoints(...)` est l'API legacy. ASP.NET Core 7+ recommande `app.MapXxx` directement.
- **Recommandation :** Migrer vers `app.MapGroup` direct (sans `UseEndpoints`).

### API-008 — Aucun endpoint ne retourne `Cache-Control` / `ETag`
- **Sévérité :** low
- **Statut :** NEW
- **Constat :** Toutes les listes retournent sans cache. Sur des reads fréquents (`ListMyProjects`, `ListProjectResources`) c'est dommage.
- **Recommandation :** `ResponseCaching` + ETag sur les Get.

### API-009 — `IMapper` injecté partout (couche Api fait du mapping)
- **Sévérité :** low
- **Statut :** NEW
- **Constat :** Les controllers font `mapper.Map<Response>(domainResult)`. Le mapping devrait être dans Application (handlers retournent déjà des `*Result` qui pourraient être directement le DTO).
- **Recommandation :** Mapping uniquement côté Application ; Api retourne tel quel.
